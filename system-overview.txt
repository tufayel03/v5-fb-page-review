1. SYSTEM ARCHITECTURE & FILE STRUCTURE
-----------------------------------------------------------------------------
The project is a Web Application (Vite + React frontend & Node.js + Express backend) 
integrated with a specialized SQLite database and supported by a custom Chrome Extension 
for administrative scraping/ fraud monitoring.

Core Files & Roles:
* /server.ts
  Contains the backend API server (Express + TypeScript), including routing, 
  moderator validation middleware, HTML entities decoding, and Image optimization.
* /database.ts
  Sets up and manages the SQLite database, handles connection settings, and exposes
  prepared statements for high-performance WAL-mode transactions.
* /chrome-extension/
  * manifest.json: Extension configuration, permissions (activeTab, storage, scripting),
                   matches facebook.com for the content script, current version (v1.4).
  * content.js: Scrapes Facebook page name, page URL, and profile picture from the DOM.
  * popup.html: Compact, high-fidelity dark-mode SaaS UI styled to perfectly match 
                 the fbpagereview.com website theme (deep blue #050b18 & brand green #0fbc6f).
  * popup.js: Handles extension authentication, checks database status in real-time,
              populates input fields, and posts data securely to the backend API.
* /chrome-extension.zip
  The packed archive of the extension folder, ready to be extracted and loaded into Chrome.

2. DATABASE SCHEMA (SQLite in WAL Mode)
-----------------------------------------------------------------------------
Table: FacebookPages
* id (TEXT, Primary Key): Unique identifier of the tracked page.
* current_name (TEXT): UTF-8 decoded page title (e.g. including Bangla strings).
* facebook_url (TEXT, Unique): Scraped page URL (canonical identifier).
* contact_number (TEXT): Main scraped or manual mobile number.
* extra_contacts (TEXT): Comma-separated alternative contact numbers.
* payment_methods (TEXT): Comma-separated payment methods (e.g. bkash, nagad, rocket).
* page_details (TEXT): Page details and custom notes.
* status_badge (TEXT): The page's current verified status:
  - 'Reported as Fraud' (Trust Score: -100)
  - 'Verified Marketplace Seller' (Trust Score: 100)
  - 'Under Review' (Trust Score: 0)
* trust_score (INTEGER): Trust indicator between -100 and 100.
* is_fraud_listed (INTEGER): Boolean flag (1 for fraud, 0 otherwise).
* profile_picture (TEXT): WebP optimized filename located on the server (e.g. uploads/profile-id.webp).

3. CHROME EXTENSION SCRAEPR LOGIC (content.js)
-----------------------------------------------------------------------------
To avoid capturing the administrator's own top navigation profile avatar (a common FB bug),
the scraper enforces a strict main container scoping rule:

1. mainContainer Isolation:
   The script locates `document.querySelector('[role="main"]')` to ensure it only scans the
   main page content wrapper and completely ignores global header banners, chats, and top bars.
2. SVG <image> Extraction:
   Modern Facebook headers render profile avatars within SVG `<image xlink:href>` tags rather than standard `<img>`.
   The scraper scans the mainContainer for `<image>` elements containing 'scontent' with common profile dimensions
   (e.g., 'p160x160', 'p200x200', 'p320x320') and falls back to standard images if SVG is not found.
3. UTF-8 Bangla Decoding:
   Scraped titles often contain Bangla characters encoded as numeric HTML entities.
   The backend's `decodeHTMLEntities()` handles converting these safely into standard UTF-8 Bangla text,
   ensuring names like "ঊর্মি ফ্যাশন গ্যালারী বান্টি বাজার" display natively.

4. BACKEND API ENDPOINTS FOR CHROME EXTENSION
-----------------------------------------------------------------------------
The extension securely communicates with the backend via two main endpoints:

A. GET `/api/admin/chrome-extension/check-page?url=<facebook_url>`
   - Authenticated with JWT token (`requireModerator` middleware).
   - Searches the database by the URL to check if the page is already registered.
   - Returns JSON: { exists: true/false, page: { status, contactNumber, paymentMethods, pageDetails } }.
   - If the page exists, the extension's popup dynamically updates its UI:
     * Displays a glowing neon status badge (Red for Fraud, Gold for Verified, Green for Under Review).
     * Automatically pre-fills all input fields (contact number, payments, notes) with their database state.
     * Updates button actions (e.g., "⭐ Verified Seller" updates to "⭐ Update Verified", etc.).

B. POST `/api/admin/chrome-extension/add-page`
   - Authenticated with JWT token (`requireModerator`).
   - Receives payload: { facebookUrl, name, profilePictureUrl, status, contactNumber, paymentMethods, pageDetails }.
   - Validates roles: Automatically maps lowercase database roles like `'admin'` to `'Super Admin'`
     to ensure seamless extension authentication compatibility.
   - Legacy entity decoder handles converting HTML strings to pristine UTF-8 text.
   - High-Performance Image Optimization:
     If the page is new or has a new profilePictureUrl, the server fetches the image and runs
     `sharp` to resize it to `300x300` and save it directly in `uploads/` as a lightweight WebP image.
   - Trust score mapping:
     * 'Reported as Fraud' -> trust_score = -100, status_badge = 'Reported as Fraud'
     * 'Verified Marketplace Seller' -> trust_score = 100, status_badge = 'Verified Marketplace Seller'
     * 'Under Review' -> trust_score = 0, status_badge = 'Under Review'

10. DUAL-MODE PROFILE PICTURE MANAGEMENT (SYNC vs UPDATE)
-----------------------------------------------------------------------------
* Multi-Tier Profile Picture Scraper:
  The crawler pipeline follows a four-tier fallback structure:
  1) Public Page Plugin widget iframe -> 2) Direct Page HTML -> 3) Google Translate Proxy -> 4) Graph API redirect.
* Tiered Validation (Sharp Integration):
  Every fallback tier immediately validates downloaded content using `sharp().metadata()`. If Facebook or the CDN blocks the request and returns an invalid format (like a 403 Forbidden HTML error page instead of a binary image), the system automatically deletes the corrupt file and falls back to the next tier without throwing/crashing.
* Sync Mode (Auto-Fetch Profile Pictures):
  Exclusively targets page records with no existing profile picture (where the database lists the picture as NULL, "", or "failed"). Skips pages that already have valid pictures.
* Update Mode (Update Profile Pictures):
  Forced refresh of existing images. Fetches the latest profile picture from Facebook, automatically deletes both the old full-sized WebP and old thumbnail WebP files from the server's uploads/ folder to prevent storage bloat, and updates the database record.

11. AUTOMATIC FACEBOOK SHARE SHORTLINK RESOLUTION
-----------------------------------------------------------------------------
* Shortlink Redirect Resolution:
  Users searching/adding pages using Facebook share shortlinks (e.g., `https://www.facebook.com/share/18Dhxpq5Je`) are supported natively.
* Execution Flow:
  - The backend detects `/share/` in the input URL query.
  - Resolves the 302 location header using a lightweight curl request.
  - Automatically cleans parameters from the resolved location URL (stripping `?rdid=...` and `#...`).
  - The central scraper (`scrapeAndAddFacebookPage`) continues using the resolved canonical page URL, avoiding wrong shortlink usernames as names, and seamlessly matching existing page records in the database.
